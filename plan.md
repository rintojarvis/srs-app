# 暗記アプリ（強制復習・SRS）方針整理

作成日: 2026-05-15
リマインダー元: 「忘却曲線ベースの強制復習アプリ作成（Anki風暗記カード・iPhone対応・未回答だと操作ロック）」

---

## 結論（先に1段落で）

**「真の OS レベル操作ロック」は個人開発では現実的に不可能**。Family Controls は配布承認がボトルネックで、しかも自分で iOS 設定から剥がせる。よって本案件は「**FSRS アルゴリズム × iPhone PWA × ソフトな強制（通知連打 + Shortcuts + UX）**」を本線とし、ロックの弱さが運用で本当に致命的だと実証できたときだけ Mac + ネイティブに進む、という二段ロケットで設計する。Mac/Apple Developer/Swift 学習を最初から組み込むのは試験までの時間に対して投資効率が悪い。

---

## 1. 構成（推奨：PWA + FSRS）

| 層 | 採用 |
|---|---|
| アルゴリズム | **FSRS-5**（評価を Again/Good の 2 択に縮約） |
| クライアント | **PWA**（Safari + ホーム画面追加、IndexedDB、Service Worker、Web Push） |
| 強制復習 | Web Push 通知連打 + 巨大バッジ + Shortcuts オートメーション（Safari/SNS 起動時に SRS PWA へ遷移） |
| データ | ローカル（IndexedDB）。PC との同期は後回し or 単純な JSON エクスポート |
| 開発機材 | DELL（Windows）だけで完結 |

代替案（明示的に却下）:
- ネイティブ iOS：Mac + USD 99/年 + Family Controls 承認待ち → 半年運用してロック不足が実証されてから検討
- Leitner system：1,000 枚規模では破綻
- Web アプリ（ホスト型）：オフラインで弱い、iPhone 連携が浅い

### 確定済み（追加・2026-05-15 第3ラウンド）

| 論点 | 決定 |
|---|---|
| 最初の入力ソース | **授業後まとめ MD**（既存ノートが豊富、評価しやすい） |
| カードスキーマ | **フレキシブル JSON**（タグで任意フィールド可） |

### 全体設計（v1 確定スコープ）

**入力**: 授業後まとめ MD ファイル（手元の既存ノート）

**処理パイプライン**:
1. ユーザーが MD ファイルを `srs-app/inbox/` に投入
2. Claude（`claude -p` または Agent ツール）が MD を読んで FSRS カード JSON を生成
3. カード JSON が Cloudflare D1 に同期される
4. PWA から復習開始

**復習画面の入力 4 項目**（毎回の復習で取る）:
1. **質問**（front 表示）
2. **回答**（back 表示・自分の答えとつき合わせ）
3. **習熟度評価**（FSRS 4 択: Again / Hard / Good / Easy）= 自分がどれだけできてきたか
4. **カード自体への評価**（★この問題自体への AI 向けフィードバック）
   - 選択肢ボタン：`このまま` / `不要・削除` / `もっと深掘り` / `分割すべき` / `表現を直して` / `論点ズレてる`
   - **自由テキスト入力欄**（任意・1〜2 行）：「この聞き方じゃなくて○○を聞くべき」「ここはいらない」等を即時に書ける

**カード JSON 例（フレキシブル + 評価ログ蓄積）**:
```json
{
  "id": "card_001",
  "front": "民法177条の『第三者』とは？",
  "back": "当事者及びその包括承継人以外の者で、登記の欠缺を主張するにつき正当な利益を有する者",
  "tags": ["民法", "物権", "対抗要件"],
  "source": "class_notes/minpou_2026-05-12.md",
  "linked_cards": ["card_015", "card_022"],
  "fsrs": { "difficulty": 0, "stability": 0, "due": "2026-05-15" },
  "review_history": [
    {
      "at": "2026-05-15T10:00:00+09:00",
      "rating": "Hard",
      "card_review": "もっと深掘り",
      "comment": "判例の射程まで聞きたい"
    }
  ]
}
```

**AI 自動進化ループ（v1 から組み込む。これが本案件の本質）**:

1. 復習のたびにカードごとの `card_review` と `comment` が `review_history` に蓄積される
2. 週1回（or カード10枚毎）、Claude にカード群と評価ログをまとめて投入
3. Claude が以下を自動実行:
   - `不要・削除` が多いカード → 削除候補リスト
   - `もっと深掘り` → 関連カードを新規生成
   - `分割すべき` → 1枚を 2〜3 枚に分割
   - `表現を直して` + comment → front/back を書き換え
   - `論点ズレてる` + comment → 論点を再特定してカード再生成
4. 変更案を `srs-app/proposals/YYYY-MM-DD.md` に出力 → ユーザーが承認したら反映

**演習連動フィードバック（v1 から組み込む）**:
- 演習・答練・模試で間違えた論点を入力する画面を用意
- 入力方法: (a) 自由テキストで「○○の論点を間違えた」と書く / (b) 既存カードを検索してチェックボックスで「これ間違えた」を選ぶ
- 入力されたら:
  1. Claude が論点キーワードを抽出 → 同じ `tags` / `source` を持つカードを検索
  2. ヒットしたカードを **優先キュー先頭**にプッシュ（次回起動時に最優先で出題）
  3. ヒットがなければ、Claude が論点に対応する新規カードを自動生成して投入
- 演習エントリは `srs-app/mistakes.jsonl` に保存され、`review_history` と同じく自動進化ループで参照される

**入力ソース拡張（v1 で全部入れる）**:
- 授業後まとめ MD（既定）
- 伊藤塾「論述マスター」テキスト → 章単位で投入 → 論点 / 規範 / 趣旨 / 関連判例の 4 枚 × N に展開
- 判例 PDF → 事案 / 判旨 / 争点の 3 枚 × N
- 六法 / 条文 CSV → 条文番号指定で条文本文 + 趣旨 + 類似条文に展開
- 投入インターフェースは `srs-app/inbox/` フォルダに置くだけ。Claude が拡張子・冒頭メタデータでソース種別を自動判定して適切なテンプレに振り分け

**ロック（v1）**:
- Web Push 通知（未復習カード N 枚あります）
- 大バッジ表示
- Shortcuts オートメーション設定の手順書を同梱

**同期（v1）**:
- Cloudflare D1（SQLite フル機能、無料枠 5GB / 25M reads/月で個人利用は永久無料圏内）
- Cloudflare Workers でカード CRUD API
- 認証は Cloudflare Access Magic Link（メール認証、SSO 不要）

### 次の具体的ステップ（実装前に1個だけ）

**サンプル変換の事前検証**: 既存の授業まとめ MD を 1 ファイル選んで、Claude にカード JSON 生成させてみる → 出力品質を見て、プロンプト設計とスキーマを微調整 → 問題なければ実装着手

### Q1. カードの中身は何か
- (a) 法科大学院の暗記事項のみ（条文・判例・趣旨フレーズ）
- (b) (a) + 自由カード（任意の暗記もの全部入れる）
- (c) その他（例：英単語、医学知識など別目的も混ぜる）

**設計影響**: (a)/(b) なら法学テンプレ（条文 → 趣旨 → 関連判例）をプリセット化できる。(c) なら汎用 UI に振る。

### Q2. 評価入力の粒度
- (a) **2 択（Again / Good）**：即答できれば Good、詰まったら Again。**強制復習との相性が最良**
- (b) 4 択（Again / Hard / Good / Easy）：Anki 標準。判断の手間が増える
- (c) タイマー型：5 秒以内に答えられれば Good、超えたら Again（強制復習らしさ最強）

### Q3. 「ロック」をどの強さで実装するか
- (a) **ソフト**：Web Push 通知 + 大バッジ + Shortcuts で SNS 起動時に SRS へ遷移
- (b) **中間**：朝の起動で「ガイド付きアクセス（Guided Access）を ON にしてください」と PWA が促す → ユーザーが手動 ON → 終わったら手動 OFF
- (c) **強硬**：将来的に Mac + ネイティブ + Family Controls で実装（時間とお金が要る）

### Q4. データ同期
- (a) ローカルのみ（iPhone 1 台運用）
- (b) PC と JSON 手動エクスポート/インポート
- (c) クラウド同期（Cloudflare KV / Supabase 等。実装増える）

### Q5. カード作成元
- (a) 手入力で 1 枚ずつ
- (b) Markdown / CSV から一括インポート
- (c) Claude にカード生成させる（既存教材 PDF を渡して FSRS カードに自動変換）

→ 法科大学院の暗記なら **(b) + (c) の併用**が圧倒的に効率的（既存の予習課題 MD やレジュメから生成できる）。

## 3. 最初の 2 週間でやる工程（PWA 第一案前提）

1. **Day 1-2**：プロジェクト雛形（Vite + TypeScript + Dexie.js for IndexedDB）と `manifest.json`、Service Worker、ホーム画面追加の動作確認（iPhone 実機）
2. **Day 3-4**：FSRS-5 を `ts-fsrs` で組み込み。カードの CRUD と「今日のキュー」生成
3. **Day 5-6**：評価入力 UI（Again/Good の 2 択）と次回出題日更新の閉ループ
4. **Day 7**：Markdown / CSV インポート機能
5. **Day 8-9**：Web Push 通知（VAPID 設定、未回答カード数を通知）
6. **Day 10**：Shortcuts オートメーションで Safari/X 起動時に PWA を開く設定の手順書を作る
7. **Day 11-12**：実データ（条文 100 枚 + 民法判例 50 枚）で 1 週間運用してログ取得
8. **Day 13-14**：ロック強度の評価。足りなければ Q3 (b)→(c) に進む判断

## 4. 既知のリスクと回避策

| リスク | 回避策 |
|---|---|
| iOS の PWA は「数週間放置で IndexedDB が消える」 | 起動時に必ずバックアップ JSON をエクスポート/ダウンロードする UI を入れる。クラウド同期に格上げする道も残す |
| Web Push 通知は iOS 16.4+ かつホーム画面追加済が条件 | 動作確認を最初に実機でやる。だめなら通知設計を作り直し |
| FSRS の真価は学習履歴 1,000 件以上で出る | 初期 100〜200 枚は SM-2 と差が出にくい点を覚悟。重みの再最適化は履歴蓄積後 |
| 「Again/Good 2 択」は FSRS の Difficulty 推定に若干影響 | 既定パラメータで十分動く。半年後に必要なら 4 択に拡張 |
